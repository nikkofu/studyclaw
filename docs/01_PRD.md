# StudyClaw 第一阶段产品需求文档

本文档用于收口 StudyClaw 第一阶段必须交付的真实需求，替代早期偏愿景化、偏远期设想的 PRD 写法。目标不是继续堆未来概念，而是把“家长布置作业 -> AI 解析 -> 孩子执行 -> 家长看到结果”这一条主链路做成稳定可演示、可发布、可协同开发的产品版本。

适用日期：

- `2026-03-09`

当前版本基线：

- 当前已交付基线：`v0.1.1`
- 下一阶段目标版本：`v0.2.0`

## 1. 第一阶段目标

第一阶段只做一件事：

让家长能够把老师当天布置的作业转成孩子可执行的任务板，并让孩子在 Pad 上独立完成、系统自动统计结果、家长及时看到反馈。

第一阶段必须交付的 7 类能力如下：

1. 家长设置今天老师发布的作业任务
2. AI 自动解析任务，并按分组 / 科目添加到孩子当天任务
3. 孩子在 Pad 上逐项完成并勾选确认
4. 每日任务完成情况形成统计结果，并及时同步给家长
5. 中文 / 英语单词默写支持“家长给清单，Pad 逐个播放”
6. 家长可按作业完成情况或校内表扬 / 批评给予 `+/-` 积分，积分在家长端和孩子端都可见
7. 日 / 周 / 月数据可视化，并由 AI 输出积极、正向、可读的学习表现分析

## 2. 用户角色

### 2.1 家长

家长是第一阶段的主要控制方，负责：

- 输入老师原始作业
- 审核 AI 解析结果
- 发布当天任务
- 查看完成进度、统计和报告
- 录入单词清单
- 进行积分奖励或扣减

### 2.2 孩子

孩子是第一阶段的主要执行方，负责：

- 打开当天任务板
- 逐项完成任务
- 进行勾选确认
- 使用单词播放功能进行听写或背诵训练
- 查看自己的积分和简化版成长反馈

### 2.3 系统

系统负责：

- 解析作业文本
- 管理任务状态
- 聚合统计数据
- 生成日报 / 周报 / 月报
- 输出正向鼓励文案

系统在第一阶段不负责替家长做高风险决定，也不负责代替孩子排序学习优先级。

## 3. 第一阶段核心业务流程

### 3.1 家长布置当天作业

标准流程应为：

1. 家长选择孩子和目标日期
2. 家长粘贴老师原始作业文本，或手动补充任务
3. 系统返回解析草稿
4. 家长审核、编辑、删除或补充任务
5. 家长确认发布
6. 任务进入孩子当天任务板

设计要求：

- 必须区分“草稿”和“已发布”
- 高风险任务必须可见并支持人工确认
- 发布前允许家长手工调整学科、分组和任务标题

### 3.2 AI 解析并生成任务草稿

解析结果必须至少包含这些字段：

- `subject`
- `group_title`
- `title`
- `confidence`
- `needs_review`
- `notes`

设计要求：

- 同一主作业下的子步骤应拆成多个可勾选原子任务
- 条件任务、对象范围不明确、续做 / 订正但对象不明确的任务应标记 `needs_review`
- AI 不能直接写入正式任务板，必须经过家长确认

### 3.3 孩子在 Pad 上完成当天任务

Pad 端的核心体验是：

- 默认进入当天任务板
- 清晰展示“待完成 / 已完成”
- 支持单任务勾选
- 支持分组完成
- 支持全部完成
- 状态更新后同步回后端

设计要求：

- 首屏必须优先显示当天任务，不让孩子做复杂筛选
- 任务状态更新要有明确反馈
- 网络失败时要有重试和错误提示
- 允许刷新和重新拉取当天任务

### 3.4 家长查看完成情况和当天反馈

家长端需要能看到：

- 今日总任务数
- 已完成数 / 未完成数
- 按学科的完成情况
- 最近一次同步时间
- 今日积分变化
- 当日简版总结

设计要求：

- 第一阶段优先采用“页面拉取 + 定时轮询 + 手动刷新”实现及时同步
- 不强依赖 WebSocket 或消息推送
- 统计图表必须由确定性后端数据生成，不依赖 LLM

### 3.5 单词默写播放

第一阶段只做最小可用版：

1. 家长创建一个单词清单
2. 清单绑定到某一天、某个孩子
3. Pad 端进入“单词播放模式”
4. 系统按顺序播放单词
5. 孩子点击“下一个”继续播放

第一阶段必须有的能力：

- 顺序播放
- 重播当前单词
- 查看当前进度
- 支持中文 / 英语两类清单

第一阶段明确不做：

- 自动语音识别打分
- 拍照判卷
- 自动错词回收

### 3.6 积分与奖惩

积分需要支持两类来源：

1. 系统按任务完成规则自动结算
2. 家长手工进行 `+/-` 调整

第一阶段支持的积分来源建议固定为：

- `task_completion`
- `parent_reward`
- `parent_penalty`
- `school_praise`
- `school_criticism`

设计要求：

- 不直接修改总积分，必须保留积分流水
- 每次积分变动都要记录来源、分值、备注、时间
- 家长端展示积分流水明细
- 孩子端展示简化后的当前积分和今日变化

### 3.7 日 / 周 / 月数据与 AI 鼓励

第一阶段要让家长和孩子都能看到：

- 日视图：当天完成率、积分变化、学科分布
- 周视图：7 天完成趋势、积分趋势、学科完成分布
- 月视图：按周聚合的完成趋势和积分趋势

AI 只负责：

- 解释已经计算好的统计结果
- 生成正向、鼓励性、儿童友好的总结

AI 明确不负责：

- 直接修改统计值
- 自行决定奖惩
- 输出负面指责式文案

## 4. 第一阶段详细需求清单

### `R1` 家长发布每日任务

用户故事：

- 作为家长，我希望先看到解析草稿，再决定是否发布给孩子，避免错误任务直接进入孩子任务板。

必须支持：

- 选择 `child_id`
- 选择 `assigned_date`
- 粘贴原始作业文本
- 手动添加 / 编辑 / 删除任务
- 保存草稿
- 确认发布

验收标准：

- 家长能在 3 分钟内完成一次“输入 -> 解析 -> 审核 -> 发布”
- 已发布任务在 Pad 端当天可见

### `R2` AI 解析作业并结构化

用户故事：

- 作为家长，我希望系统先把混乱的学校群消息拆成清晰任务，但不替我做最终发布决定。

必须支持：

- 解析多学科混合原文
- 按 `学科 -> 作业分组 -> 原子任务` 输出
- 返回风险标记和说明
- 对低置信度任务允许人工编辑

验收标准：

- 主流学校群文本能自动拆成可用草稿
- 高风险任务不会被静默写入

### `R3` 孩子端当天任务板

用户故事：

- 作为孩子，我希望打开 Pad 就看到今天要做什么，并且做完一项就能立刻打勾。

必须支持：

- 默认显示当天任务
- 单任务勾选
- 分组勾选
- 全部完成
- 失败时重试

验收标准：

- 孩子无需家长协助就能完成基础勾选
- 状态同步后家长端刷新可见

### `R4` 每日完成情况与家长同步

用户故事：

- 作为家长，我希望及时看到孩子今天完成了多少，还剩多少。

必须支持：

- 当日完成统计
- 家长端刷新后可见最新状态
- 当日简报
- 最后同步时间

验收标准：

- 同一局域网或同一服务环境下，状态更新后短时间内可被家长端看见
- 当日简报不依赖人工整理

### `R5` 单词播放训练

用户故事：

- 作为家长，我希望录入英语或语文默写清单，让孩子在 Pad 上一个个听写。

必须支持：

- 清单创建
- 清单编辑
- 逐词播放
- 下一词 / 重播
- 当前进度显示

验收标准：

- 家长能独立创建一份可播放的清单
- 孩子能顺序完成一次最小播放流程

### `R6` 积分管理

用户故事：

- 作为家长，我希望对完成作业、在校表扬、批评等行为做积分调整。

必须支持：

- 自动积分规则
- 手工 `+/-` 调整
- 原因备注
- 家长端和孩子端展示

验收标准：

- 所有积分变化都能追溯来源
- 家长端和孩子端看到的积分总数一致

### `R7` 数据可视化与 AI 鼓励

用户故事：

- 作为家长，我希望看到孩子近一段时间的学习趋势；作为孩子，我希望收到积极、好理解的鼓励。

必须支持：

- 日 / 周 / 月图表
- 家长端完整视图
- 孩子端简化视图
- AI 鼓励文案

验收标准：

- 图表来自确定性统计
- AI 文案与统计一致，不胡编数据

## 5. 更好的产品建议

下面这些建议会显著提高第一阶段的可用性和后续扩展性，建议纳入设计，而不是等到代码写散后再回头补。

### 5.1 引入“草稿 / 发布”二阶段模型

不要让解析结果直接进入孩子任务板。

建议模型：

- `daily_assignment_draft`
- `daily_assignment_publish`

这样做的价值：

- 家长可审核
- 错误可修正
- 风险任务不会直接打到孩子端

### 5.2 用事件流驱动统计，而不是直接改总状态

建议把这些动作记录成事件：

- 任务创建
- 任务发布
- 任务完成
- 分组完成
- 全部完成
- 积分增加
- 积分扣减
- 单词清单播放开始 / 完成

这样做的价值：

- 日 / 周 / 月统计更稳定
- 审计和回放更容易
- 后续换存储时迁移成本更低

### 5.3 用积分流水而不是直接维护一个总分字段

建议实体：

- `points_ledger`
- `points_balance_view`

这样做的价值：

- 每次加减分都有来源
- 可以同时支持自动和手工积分
- 后续支持徽章、兑换、排行榜时更容易扩展

### 5.4 单词清单单独建模，不和普通任务完全混在一起

建议实体：

- `word_list`
- `word_item`
- `dictation_session`

原因：

- 单词播放有独立顺序、进度和播放控制
- 后续若增加“错词回放”“重播次数”“学习记录”会更自然

### 5.5 第一阶段同步策略先用“近实时拉取”

建议第一阶段使用：

- 后端写入后立即可查
- 家长端定时轮询
- 页面聚焦时自动刷新
- 提供“手动刷新”

原因：

- 比 WebSocket 简单很多
- 更适合当前团队节奏
- 足以满足第一阶段“及时同步”的要求

### 5.6 数据模型提前支持多孩子，但首版 UI 可以先优化单孩子

建议：

- 后端所有关键实体保留 `family_id` 和 `child_id`
- 首版 UI 默认围绕一个孩子展开

原因：

- 产品天然是家庭场景
- 后续支持二孩不至于推翻模型

### 5.7 明确“未完成作业的次日处理方式”

建议第一阶段至少定义规则，不一定全部实现：

- 默认仍保留在原日期
- 家长可手工转移到明天
- 报告中显示“逾期未完成”

这是作业场景里非常容易被忽略但很快会遇到的问题。

## 6. 第一阶段明确不做

以下内容先不进入 `v0.2.0`：

- OCR / 图片作业解析
- 语音识别自动打分
- 拍照判卷
- WebSocket 强实时推送
- 多智能体编排
- 情绪识别、传感器识别、坐姿检测
- 宠物养成、换装、复杂徽章体系
- 与学校系统直接打通

这些内容不是没价值，而是不应该和第一阶段的主链路争资源。

## 7. 第一阶段推荐领域模型

建议后端围绕下面这些实体收口：

- `family`
- `child`
- `daily_assignment`
- `homework_group`
- `task_item`
- `word_list`
- `word_item`
- `dictation_session`
- `points_ledger`
- `study_event`
- `report_snapshot`

其中：

- `daily_assignment` 是家长视角的“某一天任务发布单”
- `task_item` 是孩子视角的最小执行单元
- `study_event` 是统计和报告的统一数据来源
- `report_snapshot` 是日 / 周 / 月报告缓存结果

## 8. 第一阶段 Agentic 设计约束

根据 Google Cloud 关于 agentic AI design pattern 的选择建议，第一阶段采用如下约束：

- 任务解析：`custom logic pattern + single-agent system + human-in-the-loop pattern`
- 报告与鼓励：`custom logic pattern + single-agent system`
- 任务板、积分、播放控制、同步状态：不使用 Agent

原因：

- 任务解析属于“复杂语义理解 + 人工确认”的问题
- 报告属于“确定性统计 + 语言总结”的问题
- 任务状态、积分和播放控制都是高确定性业务流程，不应交给 LLM

第一阶段明确不引入：

- `multi-agent sequential pattern`
- `multi-agent parallel pattern`
- `coordinator pattern`
- `review and critique pattern`

只有当后续引入 OCR、多工具链、多角色审核或复杂内容生成时，才考虑升级。

## 9. 第一阶段成功标准

当下面条件同时满足时，可以认为第一阶段交付达标：

1. 家长能独立完成一次当天作业发布
2. AI 能返回结构化草稿，且高风险任务需要家长确认
3. 孩子能在 Pad 上独立勾选当天任务
4. 家长能看到当天完成统计和当日反馈
5. 单词清单可在 Pad 上逐个播放
6. 积分能自动和手工两种方式变动，并在双端显示
7. 日 / 周 / 月数据可视化可用，AI 能输出正向鼓励

## 10. 第一阶段之后的扩展方向

第一阶段完成后，再进入下面方向会更稳：

1. OCR / 图片作业解析
2. 听写结果自动判断
3. 错词本与复习计划
4. 推送提醒与更强实时同步
5. 更丰富的奖励系统
6. 更强的学习洞察和习惯分析
