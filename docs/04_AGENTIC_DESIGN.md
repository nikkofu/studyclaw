# StudyClaw Agentic 设计

本文档只描述当前已经落地、可维护、可测试的 Agentic 方案，不再保留和现阶段产品无关的多智能体愿景设计。

参考标准：

- `https://docs.cloud.google.com/architecture/choose-design-pattern-agentic-ai-system`

## 1. 设计原则

当前项目坚持以下原则：

1. 先确定性逻辑，后有限使用 LLM
2. 让 Agent 只处理语义理解和语言生成，不处理关键状态变更
3. 把最终创建、确认、执行留给人和业务服务
4. 所有任务状态都必须可重放、可测试、可回写到 Markdown

## 2. 当前模块与模式映射

### 2.1 作业解析模块 `taskparse`

- 主模式: `custom logic pattern`
- 辅助模式: `single-agent system`
- 辅助模式: `human-in-the-loop pattern`

为什么这样选：

- 学校群文本格式波动大，需要先做规则结构化
- LLM 只做一次受限的语义拆解，不参与长期规划
- 家长确认是必须步骤，避免错误任务直接入库

当前实现要点：

- 先提取学科、编号项、子项、条件信号
- 先生成规则兜底任务列表
- 再调用一次 OpenAI 兼容模型做语义补强
- 最终合并、去重、标记 `confidence` 与 `needs_review`

### 2.2 周报模块 `weeklyinsights`

- 主模式: `single-agent system`
- 辅助模式: `custom logic pattern`

为什么这样选：

- 周报是单次总结问题，不是复杂编排问题
- 完成数、总数等指标先由 Go 聚合
- LLM 只负责输出孩子能读懂的自然语言总结

当前实现要点：

- 读取近 7 天 Markdown 任务数据
- 统计总任务数和完成数
- 可选调用一次模型生成摘要、优势和改进建议
- 如果模型不可用，回退到 mock 洞察

### 2.3 任务板与状态同步 `taskboard`

- 模式: 不使用 Agent

为什么这样选：

- 任务板是核心业务状态，不应交给非确定性模型
- 这里的正确做法是领域服务 + 仓储
- 状态更新必须确保可预测和可回滚

## 3. 为什么不采用多智能体

当前范围内不采用 `multi-agent system`，原因很直接：

- 任务拆解和周报总结都不是需要多角色协作的问题
- 多智能体会明显增加调试、可观测性和成本复杂度
- 当前团队更需要清晰可读的业务边界，而不是智能体编排框架

只有在出现下面条件时，才考虑更复杂模式：

- 一个请求必须跨多个独立工具链协同
- 需要显式的 planner / executor / reviewer 分工
- 单次 LLM 调用已经无法满足质量和可控性

## 4. 人机协作边界

StudyClaw 当前的人机协作模型是：

1. 家长提供原始作业文本
2. 系统返回候选任务和风险标记
3. 家长确认后才真正创建任务
4. 孩子在 Pad 端自主选择和完成任务
5. 系统只同步状态和生成观察，不替代家长做关键决策

这也是 `human-in-the-loop pattern` 在当前产品里的核心落地点。

## 5. 后续扩展规则

如果未来新增新的 Agent 模块，必须先回答下面三个问题：

1. 这个问题能否先用确定性逻辑解决 80%
2. LLM 是否只负责语义理解或自然语言生成
3. 是否需要人在关键动作前确认

只有回答清楚，才允许引入新的 Agent 模式。
