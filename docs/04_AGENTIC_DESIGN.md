# StudyClaw Agentic 设计

本文档定义 StudyClaw 第一阶段及其后续近期版本中，哪些模块适合使用 Agentic pattern，哪些模块应严格保持为确定性业务服务。

参考标准：

- `https://docs.cloud.google.com/architecture/choose-design-pattern-agentic-ai-system`

## 1. 设计原则

结合 Google Cloud 的 agentic design pattern 选择建议，StudyClaw 当前坚持下面四条原则：

1. 先用确定性逻辑解决 80% 的问题，再让 LLM 处理语义理解或自然语言生成
2. 只在“需要理解原始文本”或“需要生成鼓励性总结”的地方使用 Agent
3. 所有关键状态变更都必须由业务服务负责，不允许 Agent 直接改任务状态、积分或统计值
4. 任何可能影响孩子任务结果的关键动作，都要保留人机协作检查点

## 2. Google 设计模式映射到 StudyClaw 的选择原则

根据 Google 的说明：

- 当任务是“结构化、多步骤、需要有限工具调用”的问题时，优先考虑 `single-agent system`
- 当流程存在关键人工审核点时，使用 `human-in-the-loop pattern`
- 当业务流程具有明显条件分支和强规则约束时，使用 `custom logic pattern`
- 只有当问题必须动态拆给多个专长角色时，才考虑 `multi-agent` 相关模式

StudyClaw 第一阶段的结论是：

- 任务解析需要 `single-agent + custom logic + human-in-the-loop`
- 日 / 周 / 月鼓励性总结需要 `single-agent + custom logic`
- 任务板、积分、单词播放、统计图表不适合使用 Agent

## 3. 第一阶段模块与模式映射

### 3.1 作业解析模块 `taskparse`

- 主模式：`custom logic pattern`
- 辅助模式：`single-agent system`
- 辅助模式：`human-in-the-loop pattern`

为什么这样选：

- 老师原始作业文本噪声大，需要规则预解析和结构兜底
- 单次模型调用足以做语义补强，不需要多个 Agent 讨论
- 最终是否发布必须由家长确认

第一阶段约束：

- 先做学科、编号、子项、条件短语提取
- 先产出规则草稿
- 再调用一次受限的单 Agent 语义解析
- 返回 `confidence / needs_review / notes`
- 家长确认后才能正式发布

### 3.2 日 / 周 / 月报告与鼓励 `insights`

- 主模式：`custom logic pattern`
- 辅助模式：`single-agent system`

为什么这样选：

- 图表和指标必须由后端确定性计算
- LLM 只负责把统计结果转成可读总结和鼓励语
- 这不是需要多角色协作的复杂规划问题

第一阶段约束：

- 先聚合完成率、任务数、积分变化、学科分布等指标
- 再调用一次模型生成正向总结
- 如果模型不可用，回退到模板化总结
- 绝不允许模型改写统计值

### 3.3 单词播放 `dictation playback`

- 不使用 Agent

为什么这样选：

- 第一阶段需求只是顺序播放、重播和下一项
- 这属于标准的播放控制和状态记录问题
- 这里用 Agent 只会增加成本和不确定性

第一阶段约束：

- 用确定性播放列表和会话状态控制
- 可选接入 TTS，但 TTS 不等于 Agent
- 不做语音识别评分和开放式对话

### 3.4 积分结算 `points`

- 不使用 Agent

为什么这样选：

- 加分、扣分、积分总额都必须可审计
- 积分规则应由确定性业务逻辑和积分流水决定
- 让模型参与积分计算会破坏可追溯性

第一阶段约束：

- 自动积分和手工积分都写入流水
- 总积分由流水聚合得到
- AI 可以在报告里解释结果，但不能写积分

### 3.5 任务板与状态同步 `taskboard`

- 不使用 Agent

为什么这样选：

- 任务状态是产品最核心的业务真相
- 任务完成、分组完成、全部完成都必须可预测、可测试、可回放
- 这是领域服务问题，不是模型推理问题

## 4. 为什么第一阶段不采用多智能体

当前范围内不采用下面这些模式：

- `multi-agent sequential pattern`
- `multi-agent parallel pattern`
- `loop pattern`
- `review and critique pattern`
- `coordinator pattern`

原因很直接：

1. 当前主要问题不是“需要多个专家讨论”，而是“需要把业务边界收紧”
2. 第一阶段的复杂度主要来自业务流程，而不是推理链过长
3. 多智能体会增加调试、可观测性、权限控制和成本复杂度
4. 当前团队更需要稳定接口、稳定状态、稳定文档，而不是更复杂的 agent orchestration

## 5. 第一阶段人机协作边界

StudyClaw 第一阶段的人机协作流程必须固定为：

1. 家长提供原始作业文本
2. 系统返回候选任务草稿和风险标记
3. 家长审核并发布
4. 孩子在 Pad 上执行任务
5. 系统同步状态、统计结果并生成鼓励性反馈

关键要求：

- 家长是任务发布的最终确认者
- 孩子是任务完成的执行者
- 系统是结构化、同步和总结的辅助者

## 6. 未来什么时候可以升级模式

只有在出现下面这些情况时，才考虑升级到更复杂的模式：

### 6.1 引入多模态作业解析

如果未来需要同时处理：

- OCR
- 图像语义理解
- 文字补全
- 风险复核

可以考虑：

- `multi-agent sequential pattern`

原因：

- 图片解析、文本纠错、任务结构化可以形成明确顺序

### 6.2 引入复杂内容生成和自检

如果未来要做：

- 作文点评
- 复杂学习建议
- 自动复盘并多轮优化

可以考虑：

- `review and critique pattern`
- `loop pattern`

### 6.3 引入跨多个工具链的动态决策

如果未来需要动态协调：

- 教材知识库
- 错题库
- 历史表现
- 复习计划
- 家长偏好

可以再评估：

- `coordinator pattern`

在此之前，不应提前引入。

## 7. 第一阶段的工程约束

为了保证 agentic 能力可维护，所有 Agent 模块必须遵守：

1. 输入输出结构必须可测试
2. 模型不可用时必须有降级路径
3. 模型输出必须与业务状态写入解耦
4. prompt、fixture、回归测试要成套维护
5. 不允许为了“更像智能体”而把确定性流程改成 LLM 决策流程
