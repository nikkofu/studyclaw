# StudyClaw 项目路线图

本文档基于 `2026-03-09` 的仓库状态更新，目标是把产品从“联调基线”推进到“第一阶段可交付版本”。

说明：

- 下面的 `v0.2.0 / v0.2.1` 路线图内容保留为第一阶段收口时的历史规划快照。
- 当前实际可发布增强版基线已推进到 `v0.3.1`。

## 1. 当前基线

### `v0.1.0` 最小闭环已完成

已具备：

1. 家长端粘贴学校群原始文本
2. Go 后端完成 `LLM 优先 + 规则兜底` 的任务拆解
3. 家长审核并确认创建任务
4. 任务写入 Markdown 工作区
5. Pad 端拉取任务板并同步完成状态

### `v0.1.1` 联调基线已完成

已具备：

1. Go API 错误契约统一
2. Parser / weekly insight 回归测试增强
3. Parent Web 按日期创建任务与自动化测试
4. Pad 端分层、同步测试与 Web 构建验证
5. 本地 `preflight / smoke / demo / release checklist` 入口

## 2. 下一目标版本

### `P0` `v0.2.0` 第一阶段交付版

目标：

把 StudyClaw 做成真正可向家庭演示、可交付内部试用的第一阶段产品。

`v0.2.0` 必须覆盖这 7 类能力：

1. 家长设置今天老师发布的作业任务
2. AI 自动解析任务并按分组 / 科目生成孩子当天任务
3. 孩子在 Pad 上逐项完成并勾选
4. 家长可查看每日完成统计和当日反馈
5. 支持中文 / 英语单词清单逐词播放
6. 支持任务完成积分和家长手工 `+/-` 积分
7. 提供日 / 周 / 月图表和 AI 正向鼓励

版本重点：

- 草稿 / 发布二阶段
- 任务、积分、报告三条主数据链
- 家长端和 Pad 端双端完整闭环
- 继续严格限制 Agent 边界

### `P1` `v0.2.1` 第一阶段稳定化版

目标：

把 `v0.2.0` 从“能跑通”推进到“能稳定验收、能给真实家庭试用”。

重点：

- 强化真实设备联调
- 补更多学校群解析回归样本
- 固化积分和报告接口契约
- 补字词播放流程的测试和清单
- 完善家长端和孩子端的数据一致性检查

### `P2` `v0.3.0` 学习反馈增强版

目标：

在第一阶段稳定后，再增强学习反馈和练习质量。

重点：

- 错词回放
- 单词播放速度 / 重复次数控制
- 未完成任务次日迁移
- 更细的学科统计
- 更稳定的日 / 周 / 月 AI 总结

### `P3` `v0.4.0` 多模态输入版

目标：

支持图片作业、拍照上传、OCR / VLM 解析。

重点：

- OCR / VLM 接入
- 图片 + 文本混合解析
- 多模态回归测试

## 3. 当前明确后置的能力

这些能力先不进入 `v0.2.x`：

- 情绪识别
- 传感器姿态检测
- 强实时消息推送
- AI 自动判卷
- 多智能体编排
- 宠物养成和复杂游戏化系统

原因：

当前最重要的是先证明下面这个核心价值：

“老师原始作业 -> AI 结构化 -> 家长确认发布 -> 孩子独立完成 -> 家长看到结果”

## 4. 接下来三轮执行建议

### 第 1 轮：后端领域与接口冻结

目标：

- 明确 `daily_assignment / task_item / points_ledger / report_snapshot / word_list` 的接口和数据边界
- 固化家长端、Pad 端共同依赖的契约

重点：

- API 结构
- 事件与统计口径
- AI 输出字段

### 第 2 轮：家长端与 Pad 端双端补齐

目标：

- 家长端完成“发布、积分、报告、单词清单”
- Pad 端完成“任务、播放、积分、简报”

重点：

- 任务发布体验
- 孩子执行体验
- 双端一致性

### 第 3 轮：报告与演示收口

目标：

- 把第一阶段功能做成稳定演示流
- 固化脚本、文档、验收流程

重点：

- e2e 演示清单
- smoke / demo 验收
- 发布前检查

## 5. `v0.2.0` 成功标准

`v0.2.0` 是否达标，用下面这些条件判断：

1. 家长第一次使用时，可以在较短时间内发布某一天作业
2. 学校群样式的文本能被拆成学科和原子任务
3. 孩子能独立在 Pad 上完成当天任务勾选
4. 家长能看到当日完成统计和积分变化
5. 单词清单可在 Pad 上顺序播放
6. 双端都能看到当前积分结果
7. 家长端和孩子端都能看到可用的日 / 周 / 月反馈

## 6. 推荐节奏

建议按下面顺序推进：

1. 先锁定后端领域模型和接口契约
2. 再并行推进 Parent Web 和 Pad 端
3. 最后统一做报告、演示和验收收口

如果一开始就把前后端全部拉满并行，但接口和领域模型不稳定，开发速度会更慢，而不是更快。
